## Decisions

- **2026-07-15 â€” Android large-title headers are hand-rolled (`use-collapsing-header`), not native.** The classic react-native-screens Android header implements every `headerLargeTitle*` prop as a no-op, so the tab roots always got a static pinned toolbar. A real Material `CollapsingToolbarLayout` exists in screens 4.25's experimental "gamma" stack (`type: 'small' | 'medium' | 'large'` with AppBarLayout scroll flags), but expo-router's `ExperimentalStack` whitelists only `{ title, headerShown, headerTransparent, headerBackVisible }` and drops the header type, and the gamma API is explicitly unstable â€” revisit when it grows up. Until then the three tab roots hide the native header on Android and use the shared hook: a large in-content Broth title as `ListHeaderComponent` plus an absolutely positioned compact bar that crossfades in once the title scrolls under it (Hinge-style). The collapse threshold assumes the screens' `p-4` content padding; the People add action moves from `Stack.Toolbar` into the bar's `right` slot on Android.
- **2026-07-15 â€” Android tab icons use `src` + `VectorIcon` (MaterialCommunityIcons), not the `md` prop.** The `md` Material Symbols route renders from expo-symbols' bundled static font, which has no FILL variation axis â€” filled selected states only exist for a handful of legacy glyph names (`home_filled`; nothing for group/settings). MCI has outline/filled pairs for all three tabs (`account-multiple` also matches iOS `person.2` better than the three-head `group`). Both `md` and `VectorIcon` feed the same pipeline â€” glyph rendered as a 24px white template image, tinted natively by the bar's `iconColor`/`itemIconTintList` â€” so the Lapsha tab colors apply either way. The bar itself pins every Android appearance slot (`backgroundColor`/`indicatorColor`/`rippleColor`/`iconColor`/`labelStyle`) because any unset slot silently falls back to Material You wallpaper-dynamic colors.
- **2026-07-14 â€” Android overflow/sort menus are HeroUI popovers, not `Stack.Toolbar.Menu`.** The native Compose menu cannot match the app's shape, metrics, or typography, and a transparent header leaves a gray frame around it. Android uses `headerRight` plus HeroUI `Menu`; delete confirmation uses HeroUI `Dialog`. iOS keeps the native toolbar menu and Alert.
- **2026-07-13 â€” Person management lives in one toolbar menu.** The person screen's top-right menu (ellipsis) owns edit name, add/change/remove photo, and delete person; the old "Add/Change photo" caption under the avatar is gone (tapping the empty initials circle still opens the picker). Edit name reuses the EntrySheet via a `kind: 'person'` config. Facts sorting moved into the Facts section header: a SwiftUI `Menu` with an inline `Picker` gives a tap-activated checkmarked chooser on iOS (`ContextMenu` is long-press-only); Android uses a HeroUI `Menu` popover anchored to the same glyph. Delete confirms, sets an `isDeleting` flag so the derived "Person not found" state renders nothing during the pop, then navigates back.
- **2026-07-13 â€” Header/menu chrome is iOS-version-gated on the person screen.** Inline toolbar-menu groups (the divider before Delete person) are 26+-only: iOS 18 renders them as a chunky section gap, so pre-26 gets the flat action list like Android. The white header flip over the expanded photo is pre-26-only: liquid-glass buttons adapt to the content behind them and wash out when forced white, so 26+ keeps Broth button tint and only the title (over the scrim, set via `headerTitleStyle`) goes white everywhere.
- **2026-07-14 â€” The iOS entry sheet skips `Form` entirely.** Every EntrySheet mode (edit name, fact, date, quick add, empty state) is a hand-laid `VStack` in the Lapsha card language â€” white `background`+`cornerRadius(14)` cards on a `presentationBackground(paper)` sheet, ink-primary save button, Noodle Gold `tint` on the date chip and include-year switch â€” with `fitToContents` on `BottomSheet` instead of detents: `Form`'s grouped styling can't take the Lapsha theme without `scrollContentBackground` fights, `'medium'` leaves dead space above the keyboard, and a hand-tuned fixed `{ height }` drifts with Dynamic Type. The sheet re-fits when the fact/date tab switches, and the compact date-picker popover floats fine above the fitted sheet. `presentationBackground` colors the whole sheet, so content needs no background fill of its own. SwiftUI `.infinity` doesn't cross the bridge â€” a huge `frame` maxWidth (100000) is the fill-container idiom. If a freshly (re)installed package throws "Property 'X' doesn't exist" for an export that exists in its source, restart Metro with `--clear`: the transform cache is keyed to the old package contents.
- **2026-07-13 â€” Person photos have a transient full-width view.** The person screen keeps a 96pt circle by default. A saved photo can expand in place by tap or a downward pull at the top, filling the width behind the native header and status bar; initials never expand. New avatar files are capped at 1536px so the full-width state stays sharp; existing 512px files remain valid but cannot regain detail without reselection.
- **2026-07-13 â€” Avatar photos are files plus a bare file-name column.** `persons.avatar` (schema v3) stores only a file name; the file lives in `<documents>/avatars/` because the iOS container path changes between installs. Every photo change writes a new UUID-named file so expo-image's URI-keyed cache can never show a stale photo; the replaced file is best-effort deleted after the DB write succeeds. Picking uses the system library picker with `allowsEditing` square crop (no permission prompt on either platform); `saveAvatarFile` downsizes to a 1536px maximum edge for the person screen's full-width presentation. Add-person holds the picked temp asset in state and persists it only after the person row exists (same best-effort stance as the birthday). Note the new expo-file-system API split: `File.move()` is async while `moveSync`/`delete()`/`create()` are sync â€” sync helpers must use `moveSync`.
- **2026-07-12 â€” Person rows share the timeline's visual language.** Dates use a day/month block plus `formatDateDetail`; facts use label-over-value rows. Dates have one UI order (birthday pinned, then date added), while only facts expose created/modified sorting. The native title remains the person's name rather than turning the screen into a social-style profile.
- **2026-07-02 â€” Add person is platform-native.** iOS uses a native modal and `Stack.Toolbar`; Android uses the same route as a transparent host for a HeroUI sheet because React Native Screens `formSheet` rendered full-screen. Shared state/save logic lives in `use-add-person-form.ts`; the birthday picker and switch are platform-split.
- **2026-06-10 â€” Uniwind (Tailwind v4) replaced NativeWind.** Theme variables live in `global.css`; Metro uses `withUniwindConfig` with `polyfills: { rem: 14 }`. Removing that polyfill silently enlarges rem-based sizing. `group-*` and `peer-*` are Uniwind Pro-only and otherwise no-op.
- **2026-06-10 â€” Android uses HeroUI Native; iOS uses SwiftUI-native controls.** `HeroUINativeProvider` is Android-only. Entry sheets are platform-split, with form/save behavior shared in `use-entry-form.ts`; `entry-sheet.tsx` remains a type-only shim because TypeScript does not resolve Metro platform suffixes.
- **2026-06-10 â€” HeroUI owns `accent` and `muted`; Lapsha owns the neutrals.** Overrides must follow HeroUI's import in `global.css`. Set brand amber on both `--color-accent` in `@theme` and raw `--accent` / `--accent-foreground` in each theme variant, or HeroUI pressed states stay blue.
- **2026-06-09 â€” `sort_order` is reserved for future manual ordering.** Schema v2 backfilled it for facts and dates, but the current UI does not read it.

## Gotchas

- **2026-07-15 â€” `@expo/ui` TextField still hard-crashes iOS 18 on backspace in SDK 57; patched locally.** SwiftUI's selection-enabled `TextField(selection:)` â€” which `@expo/ui` uses on iOS 18+ â€” fatally asserts (`String index is out of bounds`, `UIKitTextInput.uiSelectedRange.setter`) when text shrinks under a stale selection: fast software-keyboard backspacing in any field prefilled through the `text` observable (edit name, edit fact/date) kills the app. Upstream expo/expo#47434 was closed by merged #47447, but the clamped-binding fix still does not prevent Lapsha's fast-backspace path in unpatched `@expo/ui` 57.0.6. `patches/@expo%2Fui@57.0.6.patch` raises the availability gate in `ios/TextFieldView.swift` so only iOS 26+ (which clamps natively) gets the selectable variant; `selection`/`onSelectionChange`/`ref.setSelection` silently no-op below 26 (unused by Lapsha). Native change: requires `expo run:ios` rebuild, not Fast Refresh; re-check the patch when bumping `@expo/ui`. During diagnosis, JS-side fixes (stable modifier identities, wrapper vs field modifiers, Form vs VStack hosting, fixed detent vs `fitToContents`) all failed â€” the crash is entirely native. Hardware-keyboard backspaces rarely trigger it; rapid software delete-key taps do.

- **2026-07-13 â€” Transparent person headers need measured, platform-specific overlap.** Read the native stack height with `useHeaderHeight` from `expo-router/react-navigation`; do not reconstruct it from safe-area insets plus 44/56. iOS content still needs a negative header-height overlap to reach the top edge, while Android edge-to-edge content already starts at `y = 0` and must not receive that offset. The hard-coded/unconditional version left an 8pt strip on iOS 26 and shifted Android content under the header.
- **2026-07-13 â€” SDK 56 precompiled Expo modules must agree on patch versions.** iOS Expo modules ship as precompiled binaries (`EXPO_USE_PRECOMPILED_MODULES` defaults on in the generated Podfile). Adding a new expo package while the lockfile holds an older `expo-modules-core` patch produces a dyld "Symbol missing" crash at launch (SIGABRT before JS) that survives clean prebuilds â€” the binary is downloaded, never compiled locally. Fix with `bunx expo install --fix` to align every expo-* patch, then rebuild; also reinstall the app on every simulator, since a stale install keeps crashing and looks identical to a failed fix. Don't `bun update expo-modules-core` â€” bun adds it as a direct dependency at the next major.
- **2026-07-13 â€” `expo run:ios` hangs silently when several simulators are booted.** With more than one booted simulator it waits on an interactive device prompt that never renders through a pipe. Pass `--device <udid>` when scripting.
- **2026-07-12 â€” Keep person-row text outside Android swipe animation nodes.** Under Fabric, translating the row tree made text selectable or disappear and could leave a stale gesture graph after Fast Refresh. `entry-row.tsx` keeps content stationary and reveals a fixed 80px action with `Gesture.Pan`. Force-stop the development build after changing gesture implementations; HMR is not a reliable verification pass.
- **2026-07-12 â€” `Stack.Toolbar` menus have Android-specific traps.** Inline submenu titles disappear and leave a stray divider, and items are hard-tinted with the toolbar tint; as of 2026-07-14 Android skips `Stack.Toolbar.Menu` entirely (see the HeroUI popover decision). Still true on iOS: the menu rejects fragments, so pass an array of `MenuAction`s. See `components/person/person-menu.ios.tsx`.
- **2026-07-02 â€” A route-mounted HeroUI sheet needs a retried open and a recoverable close.** Start closed and request open after the content's first `onLayout`, then retry the Gorhom ref's `snapToIndex(0)` until `onAnimate` starts: HeroUI's one transition-driven call can run before Gorhom's container/handle layout is ready and is silently dropped. Ignore the initial spurious `onOpenChange(false)` only while `isOpen` is still false; once open was requested, every close event must pop the transparent route even if animation never began, or the invisible overlay traps all input. Keep the deep-link fallback `canGoBack() ? back() : replace('/')`. See `add-person-sheet.android.tsx`.
- **2026-07-01 â€” Android keyboard avoidance belongs to `react-native-keyboard-controller`.** Drive resizable sheet padding from `useReanimatedKeyboardAnimation()` (`height` is negative while open), and start autofocus shortly after the sheet's `onAnimate` begins so sheet and keyboard motions overlap. Gorhom's keyboard height resolves to zero under the root `KeyboardProvider`. `softwareKeyboardLayoutMode: "pan"` is ignored on Android 15+ but can double-shift manual padding on Android 14 and below; retest there if those versions become targets.
- **2026-06-10 â€” Light mode requires two locks.** Keep `Appearance.setColorScheme('light')` and `Uniwind.setTheme('light')` together in `app/_layout.tsx`; cold starts can otherwise render HeroUI dark even when hot reload looks correct.
- **2026-06-10 â€” Shared icons cannot use `expo-image` `sf:` sources.** Android Glide silently rejects them. Use `components/ui/icons.tsx`; Android `Stack.Toolbar` icons require `require()`d XML drawables.
- **2026-06-10 â€” Do not put HeroUI `Select` inside the entry sheet.** Its popover, dialog, and nested-sheet presentations all failed on-device. The inline expanding person list is deliberate.
- **2026-06-10 â€” HeroUI BottomSheet requires a false-to-true open transition.** Keep it mounted closed and gate its content; mounting it already open can leave it permanently closed without an error.
- **2026-06-10 â€” M3 `DatePickerDialog` returns UTC-midnight milliseconds.** Read with `getUTC*()` to avoid a one-day shift in western timezones. Every Jetpack Compose view, including dialogs, must be a direct child of `<Host>`.
- **2026-07-13 â€” Disabled native-tab taps use Expo Router's `tabPress` listener.** Expo Router 56.2.14 owns `onTabSelectionPrevented` and re-emits the prevented selection as `tabPress`. Attach quick add through the disabled trigger's `listeners`; a host callback passed through `NativeTabs.unstable_nativeProps` is overwritten internally.
- **2026-06-10 â€” SwiftUI may reuse TextFields across the fact/date switch.** If placeholders become stale, give the two tab bodies distinct native view identities.
- **2026-06-09 â€” Stamp `PRAGMA user_version` inside each migration transaction.** A Fast Refresh once stamped v2 without applying it. `migrate()` still self-heals that exact schema mismatch; keep the repair until old development databases no longer matter.
