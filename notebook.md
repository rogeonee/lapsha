## Decisions

- **2026-07-13 — Person management lives in one toolbar menu.** The person screen's top-right `Stack.Toolbar.Menu` (ellipsis) owns edit name, add/change/remove photo, and delete person; the old "Add/Change photo" caption under the avatar is gone (tapping the empty initials circle still opens the picker). Edit name reuses the EntrySheet via a `kind: 'person'` config. Facts sorting moved into the Facts section header: a SwiftUI `Menu` with an inline `Picker` gives a tap-activated checkmarked chooser on iOS (`ContextMenu` is long-press-only); Android uses the avatar-menu Alert pattern. Delete confirms, sets an `isDeleting` flag so the derived "Person not found" state renders nothing during the pop, then navigates back. Android paths (menu, Alert chooser, sheet name mode, `more_vert.xml` trigger) are not yet device-verified.
- **2026-07-13 — Header/menu chrome is iOS-version-gated on the person screen.** Inline toolbar-menu groups (the divider before Delete person) are 26+-only: iOS 18 renders them as a chunky section gap, so pre-26 gets the flat action list like Android. The white header flip over the expanded photo is pre-26-only: liquid-glass buttons adapt to the content behind them and wash out when forced white, so 26+ keeps Broth button tint and only the title (over the scrim, set via `headerTitleStyle`) goes white everywhere.
- **2026-07-14 — Themed compact SwiftUI sheets skip `Form`.** The edit-name sheet is a hand-laid `VStack` (card-white field and ink-primary button on paper) using `fitToContents` on `BottomSheet` plus `presentationBackground(paper)` on the sheet's `Group` — not a detent: `Form`'s grouped styling can't take the Lapsha theme without `scrollContentBackground` fights, `'medium'` leaves dead space above the keyboard, and a hand-tuned fixed `{ height }` drifts with Dynamic Type. `presentationBackground` colors the whole sheet, so the content needs no background fill of its own. SwiftUI `.infinity` doesn't cross the bridge — a huge `frame` maxWidth (100000) is the fill-container idiom.
- **2026-07-13 — Person photos have a transient full-width view.** The person screen keeps a 96pt circle by default. A saved photo can expand in place by tap or a downward pull at the top, filling the width behind the native header and status bar; initials never expand. New avatar files are capped at 1536px so the full-width state stays sharp; existing 512px files remain valid but cannot regain detail without reselection.
- **2026-07-13 — Avatar photos are files plus a bare file-name column.** `persons.avatar` (schema v3) stores only a file name; the file lives in `<documents>/avatars/` because the iOS container path changes between installs. Every photo change writes a new UUID-named file so expo-image's URI-keyed cache can never show a stale photo; the replaced file is best-effort deleted after the DB write succeeds. Picking uses the system library picker with `allowsEditing` square crop (no permission prompt on either platform); `saveAvatarFile` downsizes to a 1536px maximum edge for the person screen's full-width presentation. Add-person holds the picked temp asset in state and persists it only after the person row exists (same best-effort stance as the birthday). Note the new expo-file-system API split: `File.move()` is async while `moveSync`/`delete()`/`create()` are sync — sync helpers must use `moveSync`.
- **2026-07-12 — Person rows share the timeline's visual language.** Dates use a day/month block plus `formatDateDetail`; facts use label-over-value rows. Dates have one UI order (birthday pinned, then date added), while only facts expose created/modified sorting. The 72px initial avatar is an identity marker; the native title remains the person's name rather than turning the screen into a social-style profile.
- **2026-07-02 — Add person is platform-native.** iOS uses a native modal and `Stack.Toolbar`; Android uses the same route as a transparent host for a HeroUI sheet because React Native Screens `formSheet` rendered full-screen. Shared state/save logic lives in `use-add-person-form.ts`; the birthday picker and switch are platform-split.
- **2026-06-10 — Uniwind (Tailwind v4) replaced NativeWind.** Theme variables live in `global.css`; Metro uses `withUniwindConfig` with `polyfills: { rem: 14 }`. Removing that polyfill silently enlarges rem-based sizing. `group-*` and `peer-*` are Uniwind Pro-only and otherwise no-op.
- **2026-06-10 — Android uses HeroUI Native; iOS uses SwiftUI-native controls.** `HeroUINativeProvider` is Android-only. Entry sheets are platform-split, with form/save behavior shared in `use-entry-form.ts`; `entry-sheet.tsx` remains a type-only shim because TypeScript does not resolve Metro platform suffixes.
- **2026-06-10 — HeroUI owns `accent` and `muted`; Lapsha owns the neutrals.** Overrides must follow HeroUI's import in `global.css`. Set brand amber on both `--color-accent` in `@theme` and raw `--accent` / `--accent-foreground` in each theme variant, or HeroUI pressed states stay blue.
- **2026-06-09 — `sort_order` is reserved for future manual ordering.** Schema v2 backfilled it for facts and dates, but the current UI does not read it.

## Gotchas

- **2026-07-14 — `@expo/ui` TextField hard-crashes iOS 18 on backspace; patched locally.** SwiftUI's selection-enabled `TextField(selection:)` — which `@expo/ui` uses on iOS 18+ — fatally asserts (`String index is out of bounds`, `UIKitTextInput.uiSelectedRange.setter`) when text shrinks under a stale selection: fast software-keyboard backspacing in any field prefilled through the `text` observable (edit name, edit fact/date) kills the app. The clamped-binding workaround shipped in 56.0.21 does not fully prevent it (expo/expo#47434, open). `patches/@expo%2Fui@56.0.21.patch` raises the availability gate in `ios/TextFieldView.swift` so only iOS 26+ (which clamps natively) gets the selectable variant; `selection`/`onSelectionChange`/`ref.setSelection` silently no-op below 26 (unused by Lapsha). Native change: requires `expo run:ios` rebuild, not Fast Refresh; re-check the patch when bumping `@expo/ui`. During diagnosis, JS-side fixes (stable modifier identities, wrapper vs field modifiers, Form vs VStack hosting, fixed detent vs `fitToContents`) all failed — the crash is entirely native. Hardware-keyboard backspaces rarely trigger it; rapid software delete-key taps do.

- **2026-07-13 — Transparent person headers need measured, platform-specific overlap.** Read the native stack height with `useHeaderHeight` from `expo-router/react-navigation`; do not reconstruct it from safe-area insets plus 44/56. iOS content still needs a negative header-height overlap to reach the top edge, while Android edge-to-edge content already starts at `y = 0` and must not receive that offset. The hard-coded/unconditional version left an 8pt strip on iOS 26 and shifted Android content under the header.
- **2026-07-13 — SDK 56 precompiled Expo modules must agree on patch versions.** iOS Expo modules ship as precompiled binaries (`EXPO_USE_PRECOMPILED_MODULES` defaults on in the generated Podfile). Adding a new expo package while the lockfile holds an older `expo-modules-core` patch produces a dyld "Symbol missing" crash at launch (SIGABRT before JS) that survives clean prebuilds — the binary is downloaded, never compiled locally. Fix with `bunx expo install --fix` to align every expo-* patch, then rebuild; also reinstall the app on every simulator, since a stale install keeps crashing and looks identical to a failed fix. Don't `bun update expo-modules-core` — bun adds it as a direct dependency at the next major.
- **2026-07-13 — `expo run:ios` hangs silently when several simulators are booted.** With more than one booted simulator it waits on an interactive device prompt that never renders through a pipe. Pass `--device <udid>` when scripting.
- **2026-07-12 — Keep person-row text outside Android swipe animation nodes.** Under Fabric, translating the row tree made text selectable or disappear and could leave a stale gesture graph after Fast Refresh. `entry-row.tsx` keeps content stationary and reveals a fixed 80px action with `Gesture.Pan`. Force-stop the development build after changing gesture implementations; HMR is not a reliable verification pass.
- **2026-07-12 — `Stack.Toolbar` menus have Android-specific traps.** Inline submenu titles disappear and leave a stray divider, so inline groups stay iOS-only. Android hard-tints menu items with the toolbar tint; tint the toolbar ink and the trigger amber separately. The menu rejects fragments, so pass an array of `MenuAction`s. See `screens/person/person-screen.tsx`.
- **2026-07-02 — A route-mounted HeroUI sheet needs three guards.** Start closed and open from the content's first `onLayout`; ignore the initial spurious `onOpenChange(false)` until opening begins; close with `canGoBack() ? back() : replace('/')` for deep links. See `add-person-sheet.android.tsx`.
- **2026-07-01 — Android keyboard avoidance belongs to `react-native-keyboard-controller`.** Drive resizable sheet padding from `useReanimatedKeyboardAnimation()` (`height` is negative while open), and start autofocus shortly after the sheet's `onAnimate` begins so sheet and keyboard motions overlap. Gorhom's keyboard height resolves to zero under the root `KeyboardProvider`. `softwareKeyboardLayoutMode: "pan"` is ignored on Android 15+ but can double-shift manual padding on Android 14 and below; retest there if those versions become targets.
- **2026-06-10 — Light mode requires two locks.** Keep `Appearance.setColorScheme('light')` and `Uniwind.setTheme('light')` together in `app/_layout.tsx`; cold starts can otherwise render HeroUI dark even when hot reload looks correct.
- **2026-06-10 — Shared icons cannot use `expo-image` `sf:` sources.** Android Glide silently rejects them. Use `components/ui/icons.tsx`; Android `Stack.Toolbar` icons require `require()`d XML drawables.
- **2026-06-10 — Do not put HeroUI `Select` inside the entry sheet.** Its popover, dialog, and nested-sheet presentations all failed on-device. The inline expanding person list is deliberate.
- **2026-06-10 — HeroUI BottomSheet requires a false-to-true open transition.** Keep it mounted closed and gate its content; mounting it already open can leave it permanently closed without an error.
- **2026-06-10 — M3 `DatePickerDialog` returns UTC-midnight milliseconds.** Read with `getUTC*()` to avoid a one-day shift in western timezones. Every Jetpack Compose view, including dialogs, must be a direct child of `<Host>`.
- **2026-06-13 — A disabled native-tab trigger does not emit `tabPress`.** It emits `onTabSelectionPrevented` through `NativeTabs.unstable_nativeProps`; this is how the iOS 26 detached quick-add tab opens without selecting a route.
- **2026-06-10 — SwiftUI may reuse TextFields across the fact/date switch.** If placeholders become stale, give the two tab bodies distinct native view identities.
- **2026-06-09 — Stamp `PRAGMA user_version` inside each migration transaction.** A Fast Refresh once stamped v2 without applying it. `migrate()` still self-heals that exact schema mismatch; keep the repair until old development databases no longer matter.
- **2026-06-09 — Keep the iOS quick-add sheet at one medium detent.** Autofocus expands a multi-detent sheet to its largest detent.
